# logi-canvus プロダクト仕様書

## 1. コンセプト

エンジニアが仕様・背景をテキストでまとめ、必要に応じてフロー図（画面遷移・ユーザーストーリーなど）を作図できるドキュメント型仕様策定ツール。企画担当者との意思疎通の円滑化を目的とする。

**一言で言うなら：** Notion的ドキュメントエディタ ＋ ドラッグ&ドロップ作図ブロック

---

## 2. ターゲット

- **メインユーザー：** 仕様の具体化と実装を担うWebアプリ開発エンジニア
- **ステークホルダー：** 仕様の妥当性を確認する企画担当者、レビューを行うチームメンバー

---

## 3. 典型的なユースケース

1. エンジニアが機能の仕様・背景をテキストで書く
2. 必要な箇所にフロー図ブロックを挿入し、ドラッグ&ドロップで作図する
3. 未確定・要確認箇所にアノテーション（ピン）を立てて質問を残す
4. ページURLを企画担当者と共有し、コメントを通じて合意形成する
5. 完成した仕様書をPRやレビューの資料として活用する

---

## 4. 組織・権限モデル

### 組織（Organization）

- アカウント作成時に「新規組織を作成」または「既存組織に参加」を選択
- 既存組織への参加は招待メール経由のみ
- 組織内の全メンバーが同等権限（閲覧・編集・ページ作成）

### 権限構造

```
Organization
  └── Members（全員が同等権限）
        ├── ページの閲覧
        ├── ページの編集
        ├── ページの新規作成
        └── 自分自身の組織脱退
```

### メンバー管理

- 他メンバーを除名する権限は誰も持たない（フラット権限）
- 自分自身のみ組織から脱退可能
- 脱退後も自分が作成したページ・アノテーション・コメントは残存する（削除されない）
- 組織内の最終メンバーが脱退した場合も組織レコードは削除しない（追跡性の観点で残存）
- 招待メールの宛先アドレスと異なるユーザーアカウントによる招待受け入れは拒否する

### ページ管理

- フォルダによるグルーピングが可能
- 各ページに一意のURLが発行される（例：`/org-slug/page-slug`）
- フォルダ・ページの作成は全メンバーが可能

---

## 5. 認証

- Email / Password のみ
- OAuthは対象外（MVP）

---

## 6. エディタ仕様

### 6-1. ページ構造：ブロック型

ページはブロックの積み上げで構成される。

```
[見出しブロック] ログイン機能 仕様
[テキストブロック] ## 背景
                  ユーザーが未ログインの場合...
[テキストブロック] ## フロー概要
                  以下の図のとおり3パターンの遷移がある。
[図ブロック]      ← ドラッグ&ドロップで作図
[テキストブロック] ## 補足・例外ケース
                  セッション切れの場合は...
```

**ブロック種別：**

| 種別 | 説明 |
|------|------|
| 見出し | H1 / H2 / H3 |
| テキスト | リッチテキスト（太字・箇条書き等） |
| 図ブロック | フロー図を作図する専用キャンバス |

### 6-2. 図ブロック（作図）

- 図ブロック内はドラッグ&ドロップでノードを配置・接続する
- ノード間の接続線はノードのエッジをドラッグして引く

**ノード種別：**

| 種別 | 用途 |
|------|------|
| Screen | 画面・ページ |
| Action | 処理・操作 |
| Branch | 条件分岐（if/else） |
| Start / End | フローの起点・終点 |
| External | 外部システム・API |

---

## 7. アノテーション（ピン）

Figmaのコメント機能に近いイメージ。

### 設置可能場所

- 図ブロック内の任意のノード
- テキストブロック内の任意の箇所

### ライフサイクル（コメントスレッド型）

```
1. ピンを立てる → 質問・コメントを書く
2. 他メンバーがスレッドに返信する
3. 合意が取れたら「Resolved（解決済み）」にして閉じる
```

- Resolvedになったピンはデフォルト非表示（表示切り替え可能）
- 未解決ピンは目立つ表示を維持する

---

## 8. 同時編集・コラボレーション

### 同時編集方式：楽観的ロック

- 複数人が同時編集可能
- 後から保存した側に競合を通知する
- 「あなたの変更と競合しました。どちらを採用しますか？」のマージUIを表示

### 通知

- ツール内通知のみ（メール通知はMVP対象外）
- 通知が発生するタイミング：
  - 自分が立てたピンにコメントが返信された
  - 自分がメンションされた
  - 競合が発生した

---

## 9. スコープ外（MVP対象外）

| 機能 | 備考 |
|------|------|
| Mermaid互換 | 対応なし |
| コードスケルトン生成 | 対応なし |
| 埋め込み用URL発行 | ページURLの共有で代替 |
| 活動通知メール（返信・メンション・競合） | ツール内通知のみ。招待メール（組織参加フロー）は対象内。 |
| OAuth（GitHub / Google） | Email/Passwordのみ |
| リアルタイム同時編集（CRDT） | 楽観的ロックで代替 |

---

## 10. データモデル（骨格）

```
Organization { id, name, slug }
User         { id, email, name, org_id }
Folder       { id, name, org_id, parent_folder_id }
Page         { id, title, org_id, folder_id, created_by,
               version（楽観的ロック用）, url_slug }
Block        { id, page_id, type（text/heading/diagram）,
               order, content, version（楽観的ロック用） }
DiagramNode  { id, block_id, type（Screen/Action/Branch/...）,
               label, position_x, position_y }
DiagramEdge  { id, block_id, source_node_id, target_node_id, label }
Annotation   { id, block_id, node_id（nullable）, block_offset（テキスト用, nullable）,
               created_by, resolved_at }
Comment      { id, annotation_id, body, created_by }
Notification { id, user_id, type,
               comment_id（nullable: annotation_reply/mention用）,
               page_id（nullable: conflict用）,
               read_at }
Invitation   { id, org_id, email, token, created_by, expires_at, used_at（nullable） }
```
